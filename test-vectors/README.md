# Tessera v0.1 Conformance Test Vectors

This directory contains the canonical conformance test vectors for Tessera v0.1. **A conformant verifier MUST treat every "valid" vector as VALID and every "invalid" vector with the documented failure mode.**

These vectors are mirrored to the protocol repository at `github.com/gabdim1221/veritas-protocol/test-vectors`.

## How to use

Run the conformance suite against the reference verifier:

```bash
cd veritas-verifier
npm test                         # runs all unit tests + 10 conformance vectors
```

Each vector is reproducibly generated by `scripts/generate-vectors.ts` from a fixed deterministic seed, so any third-party implementation can regenerate them and compare bit-exactly.

## Anchor verification in the suite

The conformance harness uses a **stub anchor verifier**: a proof is reported as `pending` if it base64-decodes, `invalid` otherwise. This keeps the suite fast and network-free. End-to-end Bitcoin-anchor verification is exercised separately against the live fixture at `test/fixtures/first-real-authorship.bundle.json`, which carries a real OpenTimestamps proof submitted to four calendars.

## Vector matrix

| File | Expected | Error code | Notes |
|------|----------|------------|-------|
| `authorship/valid-loa2.json` | VALID | — | LOA 2 with `webauthn` biometric attestation |
| `authorship/valid-loa3.json` | VALID | — | LOA 3 with `webauthn_liveness` |
| `authorship/valid-loa4.json` | VALID | — | LOA 4 with `kyc_bound` |
| `authorship/invalid-bad-signature.json` | INVALID | `INVALID_SIGNATURE` | First byte of signature flipped |
| `authorship/invalid-bad-anchor.json` | INVALID | `INVALID_ANCHOR` | `anchor.ots_proof` is non-base64 garbage |
| `authorship/invalid-revoked.json` | INVALID | `REVOKED` | Paired with `revocation/valid.json` via `isRevoked` lookup |
| `authorship/invalid-content-mismatch.json` | INVALID | `CONTENT_MISMATCH` | When verified with `fixtures/mismatch-content.txt` (which doesn't hash to `subject.content_hash`) |
| `delegation/valid-chain.json` | VALID | — | Stand-alone delegation signed by master key |
| `delegation/invalid-expired.json` | INVALID | `DELEGATION_EXPIRED` | When used to authorize content issued after `valid_until`. Tested via `verifyDelegation()` directly. |
| `revocation/valid.json` | VALID | — | Revocation Tessera signed by an authorized device. Targets `authorship/invalid-revoked.json` |

## Sidecar files

The `fixtures/` directory holds the original content bytes for any vector that requires content-hash verification:

- `conformance-content.txt` — the bytes that hash to the content_hash declared in the `valid-loa*.json` and `invalid-bad-signature.json` and `invalid-bad-anchor.json` and `invalid-content-mismatch.json` vectors.
- `mismatch-content.txt` — used with `invalid-content-mismatch.json` to demonstrate `CONTENT_MISMATCH`.
- `revoked-content.txt` — the bytes attested to by `invalid-revoked.json`.

## Identity

All vectors are signed under a single deterministic identity (`alice`):

- master key derives from `sha256("veritas/v0.1/test-vectors/master")`
- device key derives from `sha256("veritas/v0.1/test-vectors/device")`

This is **for fixture reproducibility only** — never use a deterministic key derivation for real-world Tesserae.

## Adding a new vector

1. Add a generator block to `scripts/generate-vectors.ts`.
2. Run `npx tsx scripts/generate-vectors.ts` from the verifier package.
3. Add a corresponding test in `test/conformance.test.ts`.
4. Update the matrix above.
5. Commit; mirror the file to `veritas-protocol/test-vectors/` (`cp -r ./test-vectors/* ../veritas-protocol/test-vectors/`).
